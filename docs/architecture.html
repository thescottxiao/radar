Decision Log

All major decisions made during the design process:

Decision Choice Rationale
Product type Multi-tenant SaaS Building for many families, not personal use
Interface WhatsApp Business API Where parents already are; group chat model fits naturally
Data sources (MVP) Gmail push + GCal webhooks + forward-to email + ICS feeds Real-time for connected accounts; forward path for privacy-cautious users
Autonomy model Auto internal, suggest external Internal actions are low-risk; external actions need human judgment
Family model WhatsApp group = tenant, N caregivers, no roles Neutral on family structure; group dynamics handle coordination naturally
Concurrent input First response wins Trusted group; speed matters more than consensus
Suggest UX Open conversation state More natural than buttons; LLM handles edit/approve classification
Onboarding Conversational, 3 exchanges max Low friction; web UI only for OAuth
Family learning Silent storage, weekly summary for correction Keeps bot helpful without feeling intrusive
Recurring events Confirm pattern once, manage autonomously One-time setup cost; ongoing benefit
Notifications Daily when actionable + weekly always + immediate for urgent High signal-to-noise ratio by default
Work calendar Deferred Valuable but complex; free/busy API adds privacy concerns
SMS ingestion Deferred to v2 Programmatically hard; forward-to-WhatsApp workaround for now
Personalization Prompt enrichment now, aggregate preference learning at scale Per-family RL has cold-start problem; rich context in prompts achieves 80% of value

System Overview

Radar is organized into four layers: Ingestion → Extraction → Reasoning → Action. The diagram below shows both components and how data flows between them:

Layer Details

Layer 1: Extraction

Email Extraction Agent
Convert raw emails into structured Event and ActionItem records
  • Tier 1 (Triage): Claude Haiku binary classification — "Is this email relevant to family/kids activities?" (~80% irrelevant, discarded immediately). Parsed with startswith("RELEVANT") for lenient matching.
  • Tier 2 (Extraction): Claude Sonnet extracts structured data into Event, ActionItem, and Learning schemas. Email content wrapped in <email_data> blocks for prompt injection defense.
  • Output: ExtractionResult containing events[], action_items[], learnings[], email_summary
  • Deduplication: Before creating Event, fuzzy-match against Event Registry: datetime ±30 min + title similarity >0.7
  • Confidence scoring: Each extraction includes 0.0–1.0 score. Below 0.6 → explicit "Is this right?" prompt. Above 0.6 → implicit correction opportunity
Calendar Change Detector
Process GCal webhook notifications and classify changes
  • Classifications: new_event, time_change, cancellation, location_change
  • Handlers: _handle_new_event() (dedup check + create), _handle_update() (diff fields), _handle_cancellation() (soft-delete)
  • Silent import: GCal changes are imported to local DB without WhatsApp notifications (user already knows)
  • ICS feed processing: Same change detection logic applied to diffed ICS feed data on each poll cycle
Intent Router (Orchestrator)
Classify inbound WhatsApp messages and route to appropriate handler
  • Pre-processing: split_compound_message() breaks multi-intent messages into single statements (Haiku, skipped for <60 char messages)
  • Classification: Button replies decoded directly (confidence=1.0). Text classified by Haiku with last 10 messages as context. Pending approval actions checked first.
  • 15 intent types: add_event, query_schedule, modify_event, cancel_event, assign_transport, release_transport, rsvp_response, share_info, approval_response, event_update, set_preference, correct_learning, general_question, greeting, unknown
  • Dispatch: route_intent() maps each intent type to a handler function in router.py or agents/calendar.py

Layer 2: Reasoning Agents

Calendar Coordinator Agent AUTO
All scheduling, conflict detection, transport coordination, and caregiver calendar synchronization
  • Event creation: handle_schedule() — LLM extracts event details → resolve datetime → dedup → create Event → link children → transport defaults → GCal outbox
  • Schedule queries: handle_query() — queries local DB first (14-day window), falls back to GCal API only if local DB is empty
  • Transport claims: handle_assignment_claim() — supports single, specific, or bulk ("all") scopes. Updates drop_off_by/pick_up_by on Event
  • Conflict detection: Before adding any event, check for overlaps. Sibling conflicts: same caregiver assigned to overlapping events (±30min) for different children at different locations
  • Event updates: Two-tier context strategy — recent conversation memory + local DB search (90-day window) + LLM matching to identify target event
Reminder Engine AUTO
Scheduled notification system with learning confirmation lifecycle
  • Daily digest: Fires ONLY when actionable (skips empty days). Content: today's events, approaching deadlines (<48h), unclaimed transport (<48h)
  • Weekly Sunday summary: ALWAYS fires. (1) Auto-confirm previously surfaced learnings, (2) graduate confirmed learnings to structured tables, (3) week-ahead events, (4) new learnings to review, (5) RSVP reminders
  • Immediate triggers: RSVP deadline <48h, unclaimed transport <48h (<4h = "TODAY" urgency tier)
  • Learning graduation: child_school → update Child.school; child_activity → append to Child.activities; contact → update ChildFriend
Onboarding Agent AUTO
Conversational 3-step family setup
  • Step 1: Extract children info (names, ages) via LLM → create Child rows, update caregiver name, send OAuth URL
  • Step 2: Wait for OAuth completion; allow skip; re-send link if still pending
  • Step 3: Set family.onboarding_complete = True, return confirmation
  • Determination: No children → Step 1; Children exist, no OAuth → Step 2; Both done → Step 3
Recurrence Detector AUTO
Pattern recognition for converting single events to recurring schedules
  • Detection: Scans 60-day window for ≥3 similar events (title similarity ≥0.7) on same day-of-week
  • Output: RecurrenceCandidate with activity_name, day_of_week, suggested RRULE, human-readable pattern
  • Trigger: Runs when new events are created; skips events already marked as recurring

Family State Store

Persistent, multi-table state hub (PostgreSQL, row-level family_id isolation)

Event Registry: All events (deduped), RSVP status, prep tasks, transport assignments (drop_off_by, pick_up_by), source_refs for GCal mapping

Family Profiles: Child data (name, age, school, grade, activities, friends), caregiver data (name, phone, Google tokens)

Conversation Memory: Short-term state (last 10–20 messages for intent context), with TTL expiry

Family Learning: Extracted facts with confidence scores and lifecycle: unconfirmed → surfaced → confirmed → graduated/superseded

GCal Outbox: Async write queue with idempotency keys. Operations: create, update, patch, delete. Exponential backoff retry.

Pending Actions: SUGGEST-mode drafts awaiting caregiver approval. Tracks action_type, draft_content, edit history, expiration.

Recurring Schedules: Activity patterns with RRULEs, transport defaults (default drop_off/pick_up caregiver), schedule exceptions.

Data Flow: GCal Sync

Three distinct sub-flows handle the bidirectional sync between Radar's local DB and Google Calendar.

A. Inbound: GCal → Radar (webhook-driven)
A user adds/edits/deletes an event directly in Google Calendar. GCal sends push notification to POST /webhooks/gcal.
api/webhooks.py gcal_webhook() → background task: _process_gcal_notification(headers, body) ingestion/gcal.py handle_gcal_notification(session, headers, body) → Extract X-Goog-Channel-ID from headers → _get_caregiver_by_channel(session, channel_id) → resource_state == "sync" → acknowledge, return (initial handshake) → fetch_calendar_changes(session, caregiver) → list[gcal_event dict] extraction/calendar.py For each gcal_event: process_calendar_change(session, family_id, gcal_event, caregiver_id) → status == "cancelled"? → _handle_cancellation() → soft-delete Event → Existing event (by source_ref gcal:{id})? → _handle_update() → diff fields, update → New? → _handle_new_event() → dedup check → create Event No WhatsApp notification sent (user already knows about their own GCal changes)
B. Outbound: Radar → GCal (outbox-driven)
Any handler that creates or updates an event enqueues a GCal write via the outbox. Never calls GCal API directly.
state/outbox.py enqueue_gcal_write(family_id, event_id, operation, payload, idempotency_key) → Idempotency key format: create:{event_id} or patch:{gcal_id}:{timestamp_ms} → Returns GcalOutboxItem or None (if duplicate key) actions/gcal_outbox_processor.py (long-running background loop) process_outbox_loop() → polls every 5 seconds → claim_pending_items(batch_size=10) (SELECT FOR UPDATE SKIP LOCKED) → For each item: operation == create → gcal.create_calendar_event() operation == update → gcal.update_calendar_event() operation == patch → gcal.patch_calendar_event() operation == delete → gcal.delete_calendar_event() → Success: mark_done() → Failure: mark_failed(error) → exponential backoff 30s → 120s → 480s → 1920s → 7200s (max 5 retries) actions/gcal.py create_calendar_event(session, family_id, event) → event_to_gcal_body(event, caregiver_map) (includes transport section) → For each caregiver with Google tokens: → GCal API .events().insert() → gcal_event_id → Store gcal:{gcal_event_id} in event.source_refs
C. Reconciliation: Periodic bidirectional sync
Cloud Scheduler triggers POST /internal/reconcile (configurable interval, default 60 min)
actions/gcal_reconciler.py reconcile_family(session, family_id) → Fetch all events from GCal API for each caregiver → Compare against local Event Registry Case 1: Local event has GCal ID but missing in GCal → Soft-delete locally (set cancelled_at) Case 2: GCal event with no local match → Dedup check → import as new Event Case 3: Both exist with discrepancies → If source=calendar: GCal wins (update local from GCal) → Else: local DB wins (enqueue outbox update to push to GCal) Returns: {created, updated, cancelled, pushed, skipped}

Data Flow: Transport Coordination

How transport assignments are determined, claimed, and tracked across events.

A. Automatic defaults (on event creation)
agents/calendar.py populate_transport_defaults(session, family_id, event) --- Gate checks (skip if ANY fail) --- → family.children is empty? → skip (no children) → event.child_id is NULL? → skip (not a child event) → len(active_caregivers) < 2? → auto-assign single caregiver silently --- Apply recurring schedule defaults --- → schedules_dal.get_active_schedules_for_family() → Match event to schedule by activity name + day_of_week → If match: apply schedule.default_drop_off + schedule.default_pick_up --- Check sibling conflicts --- → detect_sibling_transport_conflicts(session, family_id, event) → Same caregiver assigned to overlapping events (±30min) for different children at different locations → If conflict found: whatsapp.send_to_family(conflict_alert) → Conflicts flagged, not resolved. No auto-swap.
B. Manual claim (via WhatsApp)
User: "I'll do pickup for soccer" extraction/router.py classify_intent() → assign_transport route_intent() → calendar.handle_assignment_claim() agents/calendar.py handle_assignment_claim(session, family_id, message, sender_id) → [Sonnet] Extract → ExtractedBulkAssignment scope: 'single' | 'specific' | 'all' role: 'drop_off' | 'pick_up' | 'both' assigned_caregiver, assignments[] → Resolve target events (fuzzy child match + date/event hints) → _apply_single_assignment() per event → Update event.drop_off_by / event.pick_up_by → Enqueue GCal outbox update → Check sibling transport conflicts → Notify other caregivers --- Routine inference --- → Track claim: (recurring_schedule_id, day_of_week, role, caregiver_id) → After 3 consistent claims by same caregiver: → learning_dal.create_learning(category="transport_routine") → Unconfirmed. Confirmed via weekly summary (no correction = accepted). → Once confirmed: update RecurringSchedule defaults
C. Transport swap
User: "I can't do pickup Thursday" → classify_intent() → release_transport → Clear THIS event's assignment (drop_off_by / pick_up_by = NULL) → RecurringSchedule default left intact for future weeks → Notify other caregivers: "Thursday pickup for soccer is unclaimed"

Data Flow: Digests & Reminders

Three notification tiers triggered by Cloud Scheduler.

A. Daily Digest (skips empty days)
Cloud Scheduler → POST /internal/digest/daily (fires at each family's configured daily_digest_time)
api/internal.py Loop all families with onboarding_complete=True agents/reminders.py generate_daily_digest(session, family_id) → events_dal.get_upcoming_events(today) → today's events → events_dal.get_action_items_due_soon(48h) → approaching deadlines → Find events with unclaimed transport in next 48h → Nothing actionable? Return None (skip this family) → [Sonnet] Format into natural language summary → Returns str check_immediate_triggers(session, family_id) → RSVP deadline < 48h → urgent alert → Unclaimed transport < 48h → nudge → Unclaimed transport < 4h → "TODAY" urgency tier → Returns list[str] actions/whatsapp.py send_daily_digest(family_id, digest_text) → Try template daily_digest first → Fallback to free-form (if within 24h window)
B. Weekly Summary (always fires)
Cloud Scheduler → POST /internal/digest/weekly (fires on each family's configured weekly_summary_day, typically Sunday)
agents/reminders.py generate_weekly_summary(session, family_id) --- Step 1: Confirm last week's learnings --- → learning_dal.auto_confirm_previously_surfaced() → Learnings shown last week but not corrected → confirmed=True --- Step 2: Graduate confirmed learnings --- → _graduate_confirmed_learnings(family_id, learning_ids) → Promote to structured tables (Child.school, activities, etc.) --- Step 3: Gather content --- → events_dal.get_events_in_range(week_ahead) → upcoming events → learning_dal.get_unsurfaced_learnings() → new things learned → events_dal.get_events_needing_rsvp() → pending RSVPs --- Step 4: Format & send --- → [Sonnet] Format summary → learning_dal.mark_surfaced(new_learning_ids) → whatsapp.send_weekly_summary()

Suggest UX: 5 Interaction Patterns

All SUGGEST-mode actions use one of five patterns. Bot uses open conversation state — any reply is classified as edit instruction, approval, or dismissal. No buttons.

Type 1: Binary Approval
Simple yes/no decisions with no content to review
Bot: New invite: Sophia's 7th Birthday
March 28, 2-4pm @ JumpZone
RSVP needed by March 25. Add to calendar?
Caregiver: yes
Bot: Done — added to all calendars
Type 2: Content Approval
Caregiver reviews a drafted message before it goes out. Supports edit loop
Bot: Here's a draft RSVP to Sophia's mom:

"Hi! Emma would love to come to Sophia's party.
See you on the 28th! — Sarah"

Reply to edit, or say send to approve.
Caregiver: make it warmer, add that she's excited
Bot: Updated:
"Hi! Emma is SO excited for Sophia's party — she's
been counting down the days! See you on the 28th! — Sarah"
Caregiver: send it
Bot: Sent
Type 3: Options
Caregiver picks from multiple choices
Bot: Gift ideas for Sophia (turning 7, ~$30-40):

1. Melissa & Doug art kit — $35
2. National Geographic science kit — $32
3. Creativity for Kids jewelry set — $28

Reply with a number, or say "more ideas"
Caregiver: 2
Bot: Added to prep checklist: National Geographic science kit
Amazon link: [link]
Type 4: Assignment
Specific caregiver claims a task. Nudge behavior escalates if unclaimed and event approaching
Bot: Jake has soccer Saturday at 3pm (Westfield Fields)
and Emma has piano at 3:30pm downtown — same time,
different places.

Who's taking Jake? Who's taking Emma?
Caregiver 1: I'll take Jake
Caregiver 2: I got Emma
Bot: Got it — Sarah: Jake's soccer. Mike: Emma's piano.
Added to both events
Type 5: Deadline Urgency
Time-sensitive action needed. Bot surfaces link at right moment and follows up
Bot: Camp Thunderbird registration closes tomorrow.

Emma: age 9 Dates: July 7-18 Cost: $450

Want the registration link? Or I can remind you
tomorrow morning.
Caregiver: send me the link
Bot: Here: [registration link]
I'll remind you at 9am if it's not done.

Onboarding Flow

Fully conversational. Web UI exists only for Google OAuth token exchange. Completes in 3 exchanges maximum:

Onboarding requirements
  • Completes in 3 exchanges maximum

  • Forward-to email address generated at tenant creation and communicated during onboarding

Recurring Events and Season Handling

Detection

From email: If email contains explicit season schedule (e.g., "practice every Tuesday/Thursday, March 4 – May 22"), Email Extraction Agent extracts full pattern and creates RecurringSchedule record.

From calendar: Recurrence Detector observes 3+ consecutive events with same title, similar time, and same day-of-week in a 60-day window.

Confirmation

Bot: Looks like Jake has soccer practice every Tuesday and
Thursday, 4-5:30pm at Westfield Fields through May 22.
Should I track this as a recurring schedule?
Caregiver: yes
Bot: Done — Jake's soccer season tracked through May 22.
I'll manage the schedule and let you know about
any changes.

Exception Handling

  • Cancellation: Remove instance, log ScheduleException. Report in next daily digest
  • Reschedule: Update instance datetime, log exception. Check for conflicts at new time
  • Location change: Update instance location. Notify group if transport implications
  • Makeup game/event: Create new Event instance linked to schedule. Check for conflicts
  • All exceptions: Modify only the individual instance, never the overall pattern

Multi-Tenancy Design

Tenant Isolation
  • Tenant = Family = WhatsApp group. Identified by group ID
  • Database: Row-level security in PostgreSQL. Every query filters by family_id. No cross-tenant data access under any circumstance
  • Agent prompts: Include only current family's context. Never include data from other tenants in any LLM prompt
  • API keys: Per-tenant OAuth tokens stored encrypted. No shared credentials between tenants
OAuth Management
  • Each caregiver's Google refresh token encrypted at rest (AES-256, key in GCP Secret Manager)
  • Token refresh handled automatically by ingestion pipeline
  • If token expires or revoked, bot notifies that caregiver in group with re-auth link
  • Token scope: Gmail read-only + Calendar read/write
Scaling
  • Pub/Sub topics can be shared across tenants; messages include tenant ID for routing
  • GCal/Gmail watch channels are per-caregiver, tracked in Caregiver table
  • WhatsApp webhook endpoint is shared; incoming messages routed by sender phone lookup

Key Risks and Mitigations

Extraction Error Propagation
If Email Extraction Agent misidentifies event (wrong date, wrong child, false positive), family calendar becomes noisy. Mitigation: Confidence thresholds (0.6+), explicit user corrections logged as training data, weekly summary review of learnings, extraction feedback loop for model improvement
Over-automation without consent
Caregivers may feel bot is too aggressive or intrusive. Mitigation: Clear AUTO vs SUGGEST distinction, conversational approval patterns (not button-based), weekly summary for review and correction, "Pause" command to temporarily stop processing
Family dynamics conflict
Contradictory instructions from caregivers (e.g., one RSVPs yes, another RSVPs no). Mitigation: First-response-wins model works for simultaneous input; async input triggers conflict surfacing ("Sarah said yes, Mike said no — which is it?"), caregiver resolution required
OAuth token expiry cascade
If multiple caregivers' tokens expire simultaneously, bot loses ingestion capacity. Mitigation: 5-day renewal buffer (watches expire in 7 days, renewed on 5-day intervals), per-caregiver re-auth prompts, fallback to forward-to-email for token-expired caregivers
Privacy leakage via conversation memory
Conversation memory persists family data. Mitigation: Ephemeral processing (raw emails discarded immediately after extraction), conversation memory expires per TTL policy, "Delete our data" command for full tenant purge
Multi-tenancy isolation breach
Bug in agent prompt or database query leaks other family's data. Mitigation: Row-level security enforced at database level, agent prompts never include cross-tenant data, isolation tested in every deployment
Notification fatigue
Too many reminders = users mute bot. Mitigation: Daily only when actionable (skip empty days), weekly always (predictable), immediate only for urgent (conflicts, short deadlines), user can customize digest times

  • WhatsApp group bot (basic Intent Router, Calendar Coordinator)
  • Google Calendar integration (webhooks, read/write via outbox)
  • Conversational onboarding (3-step)
  • Event creation from conversation ("Add soccer practice Tuesday at 4pm")
  • Daily digest (when actionable) and weekly Sunday summary
  • Basic conflict detection
  • Transport coordination (claims, routines, sibling conflicts)
  • Forward-to email address (receive and store, basic extraction)
Phase 2 — Email Ingestion + Extraction
Validate that automatic email → structured events saves meaningful time
  • Gmail push integration (Pub/Sub)
  • Email Extraction Agent (Haiku triage + Sonnet extraction)
  • ActionItem extraction (forms, payments, items to bring)
  • Event deduplication (email ↔ calendar)
  • ICS feed subscription and attachment processing
  • Extraction feedback loop (corrections logged)
  • Family learning lifecycle (unconfirmed → surfaced → confirmed → graduated)
  • Recurrence detection (3+ similar events → RecurringSchedule)
Phase 3 — Logistics Intelligence
Differentiate from shared calendar apps. This is where Radar earns its name
  • Logistics Planner Agent (prep checklists, gear tracking)
  • Research Agent (gift suggestions, camp discovery)
  • Proactive gap detection (7 rules)
  • Full suggest UX (all 5 interaction patterns)
  • Assignment nudging with escalation
