Interface WhatsApp Business API Where parents already are; group chat model fits naturally

System Overview

Radar is organized into four layers: Ingestion → Extraction → Reasoning → Action. Each layer builds on the previous one to transform raw family data into coordinated logistics:

Layer Details

Layer 1: Extraction

Email Extraction Agent
Convert raw emails into structured Event and ActionItem records
  • Tier 1 (Triage): Claude Haiku binary classification — "Is this email relevant to family/kids activities?" (~80% irrelevant, discarded immediately)
  • Tier 2 (Extraction): Claude Sonnet extracts structured data into Event and ActionItem schemas
  • Scope: Events (parties, practices, games, appointments), action items (forms, payments, items to bring), recurring patterns, child references, contact info
  • Deduplication: Before creating Event, fuzzy-match against Event Registry: datetime ±30 min + title similarity >0.7
  • Confidence scoring: Each extraction includes 0.0–1.0 score. Below 0.6 → explicit "Is this right?" prompt. Above 0.6 → implicit correction opportunity
Calendar Change Detector
Process GCal webhook notifications and classify changes
  • Classifications: new_event, time_change, cancellation, location_change, attendee_change
  • Season awareness: If recurring instance is modified/cancelled, update only that instance in Season exceptions list
  • ICS feed processing: Same change detection logic applied to diffed ICS feed data on each 30-minute poll cycle
Intent Router (Orchestrator)
Classify inbound WhatsApp messages and route to appropriate agent
  • Intents: query, schedule, prepare, research, edit_instruction, approve, assignment_claim, update, correction, dismiss, general
  • Open conversation state: When SUGGEST-mode action is pending approval, router enters "pending approval" state. Any reply classified as edit_instruction, approve, or dismiss
  • Voice handling: Audio messages transcribed via Whisper API, then processed through intent classification pipeline

Layer 2: Reasoning Agents

Calendar Coordinator Agent AUTO
All scheduling, conflict detection, and caregiver calendar synchronization
  • Event creation: When event confirmed, create GCal events on all connected caregiver calendars
  • Conflict detection: Before adding any event, check all caregivers' calendars for overlaps. Surface conflicts to group
  • Cross-child conflicts: "Jake has soccer at 3pm and Emma has piano at 3:30pm — different locations. Who's taking whom?"
  • Playdate scheduling: Check availability → suggest times → draft message to other parent (SUGGEST) → track response
  • Season management: After confirmation, auto-create individual Event instances for each occurrence
Logistics Planner Agent AUTO
Everything that needs to happen before and around events
  • Prep checklist generation: Auto-generate ActionItems based on event type (birthday → gift + RSVP + ride check; season start → gear check + forms; camp → registration + forms + packing)
  • Gear tracking: Maintain gear_inventory on Child records. Surface gaps at season transitions: "Does Jake still have cleats that fit?"
  • Transportation planning: For recurring events, propose standing transport schedule. For one-off events, surface assignment question to group
  • Assignment nudging: If unclaimed and event approaching, nudge again. Timing is context-aware (event in 4 hours = urgent; 5 days = gentle)
Research Agent SUGGEST
External information retrieval for camps, gifts, activities, venues
  • Gift recommendations: Based on birthday child's age and family's learned budget. Present 3 options via WhatsApp. "Other ideas" re-runs with different parameters
  • Camp/activity discovery: Search for options matching child age, family location, target dates, budget. Ranked results with key details
  • Venue lookup: Party venues, activity providers, sports facilities near family's location
  • All outputs are SUGGEST mode: Require caregiver approval before any action
Reminder Engine AUTO
Scheduled notification system using cron-based evaluation + LLM for natural language
  • Daily digest: Fires ONLY when actionable (skips empty days). Content: today's events with prep status, approaching deadlines, unclaimed assignments
  • Weekly Sunday summary: ALWAYS fires. Content: week ahead, family learnings to review, season schedule updates, prep status
  • Immediate triggers: Short RSVP window (<48h), scheduling conflict detected, unclaimed transport approaching, season exception detected

Family State Store

Persistent, multi-table state hub

Event Registry: All events, deduped, RSVP status, prep tasks, transport assignments

Family Profiles: Child data (name, age, school, grade, activities, friends, gear), learned facts (contact info, schedule patterns, budget)

Conversation Memory: Short-term state for open conversation, long-term summaries with vector embeddings for semantic retrieval

Family Learning: Extracted facts from emails, calendar, conversation with confidence scores and confirmation status

Suggest UX: 5 Interaction Patterns

All SUGGEST-mode actions use one of five patterns. Bot uses open conversation state — any reply is classified as edit instruction, approval, or dismissal. No buttons.

Type 1: Binary Approval
Simple yes/no decisions with no content to review

📅 March 28, 2–4pm @ JumpZone
RSVP needed by March 25. Add to calendar?
Caregiver: yes
Bot: Done — added to all calendars ✓

Caregiver reviews a drafted message before it goes out. Supports edit loop

Reply to edit, or say send to approve.

Onboarding Flow

Fully conversational. Web UI exists only for Google OAuth token exchange. Completes in 3 exchanges maximum:

Bot: 👋 Hi! I'm Radar, your family's activity assistant.
Quick setup — what are your kids' names and ages?
Caregiver: Emma, 9 and Jake, 7
Bot: Got it — Emma (9) and Jake (7).
I'll send each person in this group a link to connect
your Google calendar and email. That's what lets me
spot events automatically.

Sarah: [OAuth link]
Mike: [OAuth link]

If you'd prefer not to connect your email, you can
also forward things to: family-abc123@radar.app
Caregiver: Done!
Bot: Both accounts connected ✓
I'll start picking up events from your inboxes.
Tell me anything else about the kids as we go —
I'll learn the rest over time.
Onboarding requirements
  • Completes in 3 exchanges maximum
  • OAuth link opens minimal web page handling Google OAuth redirect and token storage
  • Caregivers who don't complete OAuth can still participate and interact (just no email/calendar data)
  • Forward-to email address generated at tenant creation and communicated during onboarding

Recurring Events and Season Handling

Detection

From email: If email contains explicit season schedule (e.g., "practice every Tuesday/Thursday, March 4 – May 22"), Email Extraction Agent extracts full pattern and creates Season record.

From calendar: If Calendar Change Detector observes 3+ consecutive events with same title, similar time, and location, it infers recurring pattern.

Confirmation

Bot: Looks like Jake has soccer practice every Tuesday and
Thursday, 4–5:30pm at Westfield Fields through May 22.
Should I track this as a season?
Caregiver: yes
Bot: Done — Jake's soccer season tracked through May 22.
I'll manage the schedule and let you know about
any changes.

Exception Handling

  • Cancellation: Remove instance, log exception. Report in next daily digest
  • Reschedule: Update instance datetime, log exception. Check for conflicts at new time
  • Location change: Update instance location. Notify group if transport implications
  • Makeup game/event: Create new Event instance linked to season. Check for conflicts
  • All exceptions: Handled silently (no confirmation) and reported in next daily digest

Season End

Bot: Jake's soccer season ended this week. Gear to return:
shin guards (borrowed from league). Want me to archive
the season?

Implementation

Gap detection runs as part of Reminder Engine's daily evaluation job. All gap detection outputs are SUGGEST mode — they present options and ask, never auto-act.

Multi-Tenancy Design

Tenant Isolation
  • Tenant = Family = WhatsApp group. Identified by group ID
  • Database: Row-level security in PostgreSQL. Every query filters by family_id. No cross-tenant data access under any circumstance
  • Agent prompts: Include only current family's context. Never include data from other tenants in any LLM prompt
  • API keys: Per-tenant OAuth tokens stored encrypted. No shared credentials between tenants
OAuth Management
  • Each caregiver's Google refresh token encrypted at rest (AES-256, key in GCP Secret Manager)
  • Token refresh handled automatically by ingestion pipeline
  • If token expires or revoked, bot notifies that caregiver in group with re-auth link
  • Token scope: Gmail read-only + Calendar read/write
Scaling
  • Pub/Sub topics can be shared across tenants; messages include tenant ID for routing
  • GCal/Gmail watch channels are per-caregiver, tracked in Caregiver table
  • WhatsApp webhook endpoint is shared; incoming messages routed by group ID

Key Risks and Mitigations

Extraction Error Propagation
If Email Extraction Agent misidentifies event (wrong date, wrong child, false positive), family calendar becomes noisy. Mitigation: Confidence thresholds (0.6+), explicit user corrections logged as training data, weekly summary review of learnings, extraction feedback loop for model improvement
Over-automation without consent
Caregivers may feel bot is too aggressive or intrusive. Mitigation: Clear AUTO vs SUGGEST distinction, conversational approval patterns (not button-based), weekly summary for review and correction, "Pause" command to temporarily stop processing
Family dynamics conflict
Contradictory instructions from caregivers (e.g., one RSVPs yes, another RSVPs no). Mitigation: First-response-wins model works for simultaneous input; async input triggers conflict surfacing ("Sarah said yes, Mike said no — which is it?"), caregiver resolution required
OAuth token expiry cascade
If multiple caregivers' tokens expire simultaneously, bot loses ingestion capacity. Mitigation: 5-day renewal buffer, per-caregiver re-auth prompts, fallback to forward-to-email for token-expired caregivers
Privacy leakage via conversation memory
Conversation memory (embeddings, summaries) persists family data. Mitigation: Ephemeral processing (raw emails discarded immediately after extraction), conversation memory expires per policy, "Delete our data" command for full tenant purge
Multi-tenancy isolation breach
Bug in agent prompt or database query leaks other family's data. Mitigation: Row-level security enforced at database level, agent prompts never include cross-tenant data, isolation tested in every deployment
Notification fatigue
Too many reminders = users mute bot. Mitigation: Daily only when actionable (skip empty days), weekly always (predictable), immediate only for urgent (conflicts, short deadlines), user can customize digest times

Build Phases

Phase 1 — Calendar + Conversational Input (4–6 weeks)
Validate that caregivers want to interact with a WhatsApp bot for family logistics
  • WhatsApp group bot (basic Intent Router, Calendar Coordinator)
  • Google Calendar integration (webhooks, read/write)
  • Conversational onboarding
  • Event creation from conversation ("Add soccer practice Tuesday at 4pm")
  • Daily digest (when actionable) and weekly Sunday summary
  • Basic conflict detection
  • Forward-to email address (receive and store, basic extraction)
Phase 2 — Email Ingestion + Extraction (4–6 weeks)
Validate that automatic email → structured events saves meaningful time
  • Gmail push integration (Pub/Sub)
  • Email Extraction Agent (Haiku triage + Sonnet extraction)
  • ActionItem extraction (forms, payments, items to bring)
  • Event deduplication (email ↔ calendar)
  • ICS feed subscription
  • Voice note transcription
  • Extraction feedback loop (corrections logged)
Phase 3 — Logistics Intelligence (4–6 weeks)
Differentiate from shared calendar apps. This is where Radar earns its name
  • Logistics Planner Agent (prep checklists, gear tracking, transport coordination)
  • Research Agent (gift suggestions, camp discovery)
  • Season detection and management
  • Proactive gap detection
  • Family profile learning (silent + weekly summary)
  • Full suggest UX (all 5 interaction patterns)
  • Assignment nudging
Phase 4 — Growth Features (timeline TBD)
Network effects and advanced capabilities
  • Playdate network effects (two-sided scheduling when both families use Radar)
  • Crowdsourced carpooling
  • Work calendar free/busy overlay
  • SMS/iMessage ingestion
  • Budget tracking for kid activities
  • Broader home management (bills, maintenance, errands)
  • Mobile companion app (optional, WhatsApp remains primary)
